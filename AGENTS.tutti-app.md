# Prototype Design Tutti App Package

This package is generated by `pnpm package:tutti-app` from the `vibe-design` repository.

## Runtime

- `bootstrap.sh` is the only entrypoint Tutti calls.
- The server binds `TUTTI_APP_HOST:TUTTI_APP_PORT`.
- `GET /healthz` is the Tutti healthcheck endpoint.
- Durable data must stay under `TUTTI_APP_DATA_DIR`.
- Runtime scratch files must stay under `TUTTI_APP_RUNTIME_DIR`.
- Logs should go to stdout/stderr or `TUTTI_APP_LOG_DIR`.
- `bootstrap.sh` must launch Node through `TUTTI_APP_NODE`; do not call system `node`, `npm`, or `pnpm` at runtime.

## File References

- `tutti.app.json` declares `references.listEndpoint: /tutti/references/list`.
- The server implements `POST /tutti/references/list` (the Tutti Reference List protocol).
- Root level returns one navigational `group` per project (`id` = project id, `referenceCount` = exact file count under the requested `kinds`/`timeRange`).
- Entering a project group (`parentGroupId` = project id) returns its design assets as `file` references.
- Each file reference uses `location: { "type": "app-data-relative", "path": "projects/<projectId>/assets/<name>" }`. The daemon resolves the location; the app never emits absolute host paths or HTTP URLs here.
- Supported request fields: `parentGroupId`, `filterText` (direct-child filter), `limit` (clamped 1..50), `cursor` (opaque, offset-based), `timeRange` (applied to file `mtimeMs`), `kinds` (v1 is `["file"]`).
- `tutti.app.json` also declares `references.searchEndpoint: /tutti/references/search`. The server implements `POST /tutti/references/search` (the Tutti Reference Search protocol): a recursive, relevance-ranked search across every project's assets returning a flat list of `file` references with a 0..1 `score`; search never returns `group` items.
- Each search result also sets `parentGroupLabel` to the owning project's title. Because search is flattened across all projects, Tutti shows this as the result's context subtitle so users can tell which project a file belongs to. (When omitted, Tutti falls back to the app display name "Prototype Design".)

## Package Layout

- `tutti.app.json`: app manifest. The `cli.manifest` field points to `tutti.cli.json`; `references.listEndpoint` points to the file reference endpoint.
- `tutti.cli.json`: command registration for Agent GUI and Tutti CLI integrations.
- `COMMANDS.md`: human-readable command usage for the registered CLI surface.
- `server/dist/main.js`: bundled Node server entry.
- `server/dist/sql-wasm.wasm`: bundled SQLite WASM runtime asset.
- `web/dist`: prebuilt client, stylesheet, and static asset files.
- `skills`: bundled Prototype Design skill catalogue.
- User-imported skills and editable design systems are stored under `TUTTI_APP_DATA_DIR`, not the package root.

## CLI Surface

- Commands are scoped under `vibe-design`.
- Handlers are HTTP `POST /tutti/cli/*` endpoints and return Tutti CLI JSON output.
- Public CLI data commands are read-only.
- The `open` command only requests UI activation for the dashboard or an existing project.
- Project commands expose project list and project UI activation only.
- Conversation commands expose conversation indexes and messages for context lookup.
- Resource commands expose project file resource lists and resource details.
- Comment commands expose preview comment lists.
- Keep `tutti.cli.json`, `COMMANDS.md`, and the server handlers in sync when changing public Agent GUI capabilities.

Do not include `node_modules` in uploaded packages. Runtime dependencies must be bundled into `server/dist` or copied as explicit static assets during the packaging pipeline.
